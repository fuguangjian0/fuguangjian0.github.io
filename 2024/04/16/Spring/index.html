什么是Spring?

Spring是一个Java企业级应用开发框架,提供了一系列组件和工具,可以帮助开发者快速构建高效,可扩展,易维护的企业级应用,Spring的核心特性包括依赖注入(Dependency Injection)，
面向切面编程(Aspect-Oriented Programming)，控制反转(Inversion of Control).同时,Spring提供了多种插件和扩展, 如SpringMVC用于Web开发, SpringBoot用于快速
构建微服务等. Spring的成功在于他的灵活性和可扩展性, 可以根据具体需求进行定义配置和集成.

你的项目为什么使用Spring框架?

由于Spring框架具有强大的功能和良好的扩展性, 所以我的项目使用Spring框架来构建企业级应用, 具体来说, 我们使用Spring框架的依赖注入功能实现松耦合的组件设计,提高了应用的可维护性和可测试性,
同时我们还使用Spring的面向切面实现日志记录,性能监控,事务管理等横切关注点,使代码更简洁和易于维护.此外,使用SpringBoot快速构建微服务应用, 提高应用的可扩展性和部署效率.综合来说, Spring
框架为我们项目带来了很多好处, 使得我们能够更加高效的开发和维护企业及应用.

依赖注入的方式有几种?

  • 依赖注入(Dependence Injection)是一种设计模式,主要思想是通过构造器,Setter方法或接口来向对象传递依赖关系,而不是让对象自己创建或查找依赖对象,
    在Spring框架中,依赖注入是实现松耦合,可测试,易维护的重要手段,通常有以下几种手段:

  1. 构造器注入(Constructor Injection)通过对象的构造器来注入依赖对象,通常在对象创建时完成依赖注入.

  2. Setter方法注入(Setter Injection)通过对象的Setter方法来注入依赖对象, 通常在对象创建后进行依赖注入

  3. 接口注入(Interface Injection): 依赖类必须实现指定接口,然后实现接口的一个函数,该函数用来依赖注入, 该函数的参数要注入的对象, 侵入性太强,不建议使用

讲一下什么是Spring?

Spring是一个轻量级的IoC和AOP容器框架, 是为Java应用程序提供基础服务的一套框架, 目的是用于简化企业应用程序的开发, Spring的核心思想通过依赖注入和
面向切面编程来实现松耦合, 可测试和易维护的应用程序:
Spring框架提供了许多核心功能, 包括:

  1. 控制反转(Inversion of Control IOC): 将对象的创建和依赖关系的管理交给框架, 使应用程序更加松耦合.
  2. 依赖注入(Dependency Injection DI) 将对象之间的依赖关系通过注入的方式来实现, 提高了应用程序的可测试性和可维护性
  3. 面向切面编程(Aspect-Oriented Programming AOP) 通过切面来解决横切关注点的问题, 如事务控制, 安全控制, 日志记录等
  4. 数据访问层(Data Access) 提供了一套方便的API来访问数据库和其他数据存储系统, 包括JDBC, ORM, NoSQL
  5. Web开发(Web Development) 提供了一套强大的web开发框架, 包括SpringMVC, Spring WebFlux
  6. 消息队列(Messaging) , 提供了一套消息传递机制, 支持JMC, RabbitMQ等
    除此之外, Spring还提供其他功能, 如集成测试, 批处理, 缓存等

说说你对SpringMVC的理解

SpringMvc是Spring框架的一个模块, 提供了一种基于MVC(Model View Controller)架构的web应用程序开发模型, 通过控制器(Controller)管理用户请求,
通过模型(Model)来处理业务逻辑,通过视图(View)呈现响应结果, 从而实现web应用程序的分层架构和模块化开发
在SpringMVC中,用户请求首先由前端控制器DispatcherServlet接收,并将其分派给相应的处理器(Handler),处理器根据请求的URL和请求方法选择具体的处理方法, 并返回相应的
数据模型和视图名称来生成相应结果, 将其返回给用户
SpringMVC提供了许多特性,如请求映射,参数绑定,数据验证,异常处理,国际化,文件上传等,同时支持多种视图技术,如JSP Thymeleaf, Freemarker等,MVC支持Restful风格的
Web服务开发, 提供了一些Web相关的功能, 如SpringWebSockets, SpringWebFlux等

SpringMVC常用注解

用来标识处理器(Handler)和请求映射(RequestMapping)的关系的, 下面一些常用注解:

  1. @Controller 标记一个类为处理器
  2. @RequestMapping 标记处理器的方法和请求之间的映射关系, 可以指定请求的URL, 请求方法, 请求参数等
  3. @RequestParam 用于将请求参数绑定到方法参数上
  4. @PathVariable 用于将URL路径变量绑定到方法参数上
  5. @ResponseBody 用于将方法的返回值转换为HTTP响应的正文内容
  6. @RequestBody 用于将HTTP请求的正文内容绑定到方法参数上

对SpringAOP的理解

AOP(Aspect-Oriented Programming 面向切面编程) 是Spring框架的一个核心特性. 通过在程序运行期间动态将额外的行为, 如日志,安全检查等
插入到代码中, 实现更加灵活和可维护的程序设计
在Spring框架中, AOP通过以下几个概念实现:

  1. 切点(Pointcut) 指定哪些程序执行点上插入额外行为, 通常使用表达式定义切点
  2. 通知(Advice) 插入到切点的额外行为, 通常包括前置通知, 后置通知, 异常通知, 环绕通知等
  3. 切面(Aspect) 由切点和通知组成一个横切逻辑单元, 可以看作一种特殊的类
  4. 织入(Weaving) 将切面和目标对象合并, 生成一个新的代理对象, 并在运行期间插入额外行为

在Spring框架中, AOP可以应用于各种场景, 如事务管理, 缓存, 日志, 权限控制等, 可以大大简化程序设计和代码维护工作, 同时Spring框架提供各种AOP相关的注解
和工具类, 使得AOP使用更加简单方便

SpringAOP和AspectJ AOP区别

SpringAOP和AspectJ AOP都是AOP实现的, 他们可以在运行时动态为程序添加额外的行为, 但是他们之间有如下区别:

  1. 实现方式不同, SpringAOP基于JDK动态代理和CGLIB代理机制实现, 而AspectJ AOP基于字节码操纵实现
  2. 支持的切点表达式不同, AspectJ AOP支持更强大和灵活的切点表达式, 如正则表达式, 类型模式匹配, 注解匹配等. 而SpringAOP仅支持基于方法名, 参数列表, 异常类型的切点表达式
  3. 适用范围不同, SpringAOP适用于Spring框架IOC容器和SpringMVC中Web应用程序, 而AspectJ AOP是一个独立的AOP框架, 可以用于任何Java框架
  4. 集成方式不同, SpringAop可以和Spring框架的其他特性无缝衔接, 而AspectJ AOP需要通过AspectJ 编译器实现

SpringAOP 关注点和横切关注的区别

关注点: 在软件系统中, 需要关注和处理某一方面的功能模块, 例如日志, 事务, 安全等,在 AOP中, 关注点被表示为切点(Pointcut), 用于指定哪些代码位置需要执行额外行为
横切关注: 在软件系统中, 不同的关注点可能相互交叉和重叠, 例如一个方法既需要进行日志记录, 又需要事务管理, AOP中, 横切关注通常被表示为切面(Aspect), 由切点和通知组成, 用于将多个关注点的行为织入到代码中

什么是通知? 通知类型?

在SpringAOP中, 通知Advice 是指在横切关注中执行的额外代码快, 可以在方法调用前后, 异常泡出或方法执行成功后执行, 通知是实现AOP的核心机制之一, 用于将横切关注的行为织入到目标对象的方法中
SpringAOP 提供了五种类型通知:

  1. 前置通知(Before Advice) 在目标方法执行前执行的通知
  2. 后置通知(After Advice) 在目标方法执行后执行的通知
  3. 返回通知(After Returning Advice) 在目标方法执行后执行的通知, 只有方法正常返回结果时才会执行
  4. 异常通知(After Throwing Advice) 在目标方法抛出异常后执行的通知
  5. 环绕通知(Around Advice) 在目标方法执行前后都可以执行的通知, 可以控制目标方法执行的过程, 甚至可以完全取代目标方法的执行
    这些通知可以在切面中指定对应的切点, 实现对目标对象方法的拦截和处理

对IOC的理解

Spring的IOC（Inversion of Controller） 将对象的创建组装管理交给容器， 由容器负责对象之间的依赖关系。在IOC模式中，对象不再自己管理其他对象的依赖， 而是将依赖关系反转给容器进行管理。
在Spring中，IOC通过依赖注入实现的， 依赖注入是指容器通过自动或手动方式， 将对象所依赖的其他对象或资源注入到对象中， 从而达到对象之间的松耦合
通过使用IOC容器， 可以实现对象之间的解耦， 增加代码的灵活性和可维护性，同时IOC容器提供了很多功能， 例如对象的生命周期管理，AOP，事务管理等， 大大减少开发者的工作量， 提高开发效率。
总结，Spring的IOC通过依赖注入实现，将对象之间的依赖关系交给容器管理，从而实现对象之间松耦合和增强系统的可维护性和可扩展性

解释一下Spring的生命周期

  1. 实例化 Spring容器通过Bean定义中指定的构造函数或工厂方法创建bean实例
  2. 属性赋值 这个阶段，spring容器将配置文件中指定的bean属性或引用注入到bean实例
  3. 初始化 这个阶段，spring容器调用bean的初始化方法，完成bean的一些初始化方法
  4. 使用 这个阶段，bean可以被程序使用
  5. 销毁，这个阶段，spirng容器调用bean的销毁方法，完成bean的销毁操作

几种spring支持的作用域

  1. singleton 单例 整个应用程序中，只存在一个bean实例，默认情况下，spring的bean都是单例的
  2. prototype 原型 每次通过容器获取bean时，都会创建一个新的bean实例
  3. request 在一次http请求中，bean的实例是唯一的
  4. session 在一个http session中，bean的实例是唯一的
  5. global session 在一个全局http session中，bean的实例是唯一的

spring基于xml注入bean有几种方式？

  1. setter注入
  2. 构造器注入
  3. 静态工厂注入
  4. 实例工厂

spring框架用了哪些设计模式

  1. 单例模式，spring中bean默认是单例的，通过IOC容器对Bean进行管理，保证整个程序只有一个Bean实例
  2. 工厂模式，spring中IOC容器负责创建和管理Bean实例，可以将IOC容器看作一个工厂，通过工厂方法创建Bean实例
  3. 代理模式，spring的AOP基于代理实现，通过JDK动态代理或Cglib动态代理实现AOP功能
  4. 观察者模式，Spring中的事件驱动机制基于观察者模式实现，Bean可以发布事件，其他Bean可以通过监听器响应事件
  5. 模板方法模式，Spring中的JdbcTemplate和Hibernate Template等模板类，使用了模板方法模式，将重复性的操作封装到父类中，子类只需要实现特定方法即可

说说Spring中ApplicationContext和BeanFactory的区别

  1. ApplicationContext是BeanFactory的子接口，提供了更多的功能，例如国际化，事件驱动等
  2. ApplicationContext在启动时预先实例化所有的单例Bean，而BeanFactory则是按需创建Bean
  3. ApplicationContext支持Bean的自动装配，AOP等功能，而BeanFactory需要手动配置
  4. ApplicationContext提供了更多的资源加载方式，例如XML，注解，属性文件等，而BeanFactory只支持XML格式的配置文件
  5. ApplicationContext提供了更多的扩展点，例如BeanPostProcessor，ApplicationListener等，可以对Bean的生命周期和时间进行监听和处理
    如果需要使用Spring的更高级的特性，例如AOP，国际化，事件驱动等，可以选择使用ApplicationContext；如果只需要简单创建和管理Bean实例，可以选择使用BeanFactory

Spring框架的单例Bean是线程安全的吗？

在Spring框架中，默认所有Bean是单例的，整个应用只有一个Bean实例，是否线程安全，要看具体实现：

  1. 如果Bean无状态，即没有实例变量，或实例变量是线程安全的（如final修饰的常量），那Bean是线程安全的
  2. 如果Bean有状态，即有实例变量，并且这些变量非线程安全，那Bean也非线程安全。在多线程并发访问单例Bean时，出现数据竞争和并发安全问题
  3. 解决：将单例Bean的实例变量用ThreadLocal进行封装，保证每个线程访问时拥有自己的实例变量；也可以将单例Bean作用域改成原型（Prototype），每次请求都会创建一个新的Bean实例，避免并发访问问题

Spring如何解决循环依赖


整个流程：

  1. A完成初始化并曝光自己（通过ObjectFactory将自己提前曝光），在初始化时，发现自己依赖B，这时尝试get(B)，这时发现B还没创建
  2. 然后创建B，B在初始化时，发现自己依赖C，这时尝试get(C)，这时发现C还没创建
  3. 然后创建C，C在初始化时，发现自己依赖A，这时尝试get(A)，这个时候A已经添加到三级缓存中，通过ObjectFactory提前曝光，所以通过ObjectFactory#getObject()方法拿到A，C拿到A后顺利完成初始化，然后将自己添加到一级缓存中
  4. 回到B，B可以拿到C对象，完成初始化，A可以顺利拿到B完成初始化，到这里整个链路完成初始化过程

说说事务隔离级别？

  1. 读未提交(Read Uncommitted) 事务中的修改即使没有提交，也能被其他事务看到，存在脏读，不可重复读和幻读问题
  2. 读已提交（Read Committed） 事务提交后才能被其他事务看到，解决了脏读问题，但是不可重复读和幻读仍存在
  3. 可重复读 （Repeatable Read） 保证同一事务多次读取同一数据结果是一致的，解决了不可重复读问题，但幻读问题仍然存在
  4. 串行化（Serializable（））事务串行执行，可以完全解决并发问题，但是效率低
    在Spring中，可以使用@Transactional注解声明一个事务，通过设置isolation属性指定事务隔离级别，默认为Isolation.DEFAULT，表示使用数据库默认隔离级别，通常为读已提交，可以通过Isolation枚举类设置隔离级别

事务传播级别

当一个事务范围内调用另一个带有事务的方法时，当前事务如何与调用方法的事务进行交互的一种方式

  1. REQUIRED 如果当前存在事务，则加入该事务；如果当前没有事务，则创建一个新事务，这时默认传播级别
  2. SUPPORTS 支持当前事务，如果当前存在事务，则加入，如果当前没有事务，则以非事务方式执行
  3. MANDATORY 强制要求存在事务，如果存在则加入，没有则抛异常
  4. REQUIRES_NEW 创建一个新的事务，如果存在事务，则挂起当前事务
  5. NOT——WUPPORTED 以非事务方式执行，如果当前存在事务，则挂起当前事务
  6. NEVER 以非事务方式执行操作，如果当前存在事务，则抛异常
  7. NESTEDD 如果当前存在事务，则在嵌套事务内执行，如果没有事务，则执行与REQUIRED传播级别类似的操作

Spring中事务实现方式有两种：编程式事务和声明式事务

  1. 编程式事务：指通过编写代码来控制事务的提交和回滚，在编程式事务中，开发人员需要手动获取事务管理器，并在需要的地方进行事务的开启，提交，回滚等操作。编程式事务灵活，
    但是代码侵入性高，容易造成代码臃肿和难以维护，Spring推荐使用声明式事务实现事务管理
  2. 声明式事务：通过配置来控制事务的提交和回滚，在声明式事务中，开发人员只需使用注解或XML配置文件声明事务的属性和行为即可，具体的事务管理操作由框架完成
    声明式事务优点是代码简洁，易于维护，可以灵活切换事务管理方式，Spring中主要通过AOP技术实现声明式事务，其底层实现依赖于DataSourceTransactionManager和TransactionInterceptor等组件

Spring框架管理事务的优点？

  1. 简化代码：使用Spring框架观念可以大大简化代码量，使代码更易于维护
  2. 降低耦合性：使用Spring框架管理事务可以将业务逻辑和事务处理分离，降低耦合度
  3. 避免重复代码：使用Spring框架管理事务可以避免在每个业务方法中重复编写事务管理代码
  4. 支持声明式事务：Spring框架支持声明式事务，通过注解或XML配置定义事务的属性和行为
  5. 支持多种事务管理器：JDBC事务，Hibernate事务，JPA事务，JTA事务等
  6. 支持事务的回滚和提交：可以自动管理事务的提交和回滚，保证事务的一致性和完整性
  7. 提高事务的性能：使用缓存和批处理技术提高事务的性能和效率

事务注解的本质?

通过Spring AOP机制，在方法执行前后动态创建代理对象，将方法包装在一个事务中，事务注解在执行方法前开启一个事务，如果方法执行成功，则提交事务；如果方法执行过程发生异常，则回滚事务，
通过事务注解，可以在方法标记事务的属性，方便灵活控制事务的行为，同时，由于Spring框架在底层处理事务，因此可以避免一些常见的事务处理问题，如并发访问，死锁问题，事务注解使用简单，提高开发效率，降低代码复杂度
