Maven面试题

1. 什么是Maven？

Maven是开源的项目管理和构建工具，帮助开发人员自动化构建，测试和部署java项目，提供标准化项目结构和强大的构建规则，丰富的插件生态系统，使得项目的构建和依赖管理简单可靠
核心概念时项目对象模型POM，Project Object Model，坐标，依赖管理，生命周期，POM是一个xml文件，描述了项目的基本信息，依赖信息，插件配置等
依赖管理功能可以让开发人员方便的使用和管理项目所依赖的外部库和模块，生命周期定义一些列构建阶段和插件，开发人员根据需求执行不同的要求

2.Maven能解决什么问题？

  1. 添加第三方jar包：使用Maven之前，手动复制jar包到项目的WEB-INF/lib下，每个项目都有一个，造成大量重复文件，而maven将jar包放到本地仓库统一管理，需要jar包只需要用坐标引用
  2. jar包之间的依赖关系：jar包往往不独立，很多jar包在其他jar包支持下能正常工作，一个jar包会依赖另一个jar，如果手动导入容易出错，maven将当前jar包依赖的其他jar包全部导入
  3. 获取第三方jar包：每个jar包在官网获取方式不同，给工作带来麻烦，使用Maven以做标方式依赖jar包，从中央仓库下载，同时下载这个jar依赖的其他jar
  4. 将项目拆分为多个模块：如果项目很大，不能通过package结构划分，将项目拆分成多个工程协同开发

3.Maven优缺点？

优点：
简化项目依赖管理，
易于上手，新手了解几个命令即可，
便于持续集成工具(jenkins)整合，
便于项目升级，无论是项目本身还是项目使用的依赖，
maven有很多插件，便于功能扩展，如生产站点，自动发布版本等

缺点：
庞大的构建系统，学习难度大，入门容易但精通难
采用约定大于配置策略，一旦出现问题，很难调试（网络环境差，导致很多仓库无法访问）

4.什么是maven坐标？

用于标识一个项目和模块的信息，由一组固定属性组成包括groupId version artifactId
groupId 组织名称：标识项目所属的组织或团队，一般情况时翻转的域名com.bt 也可以自定义
artifactId 项目组织：用于标识项目唯一标识符，区分不同的项目，通常是项目名称，用在仓库中唯一标识一个项目，在同一个groupId下，不同项目有不同的artifactId
version 版本号：区分不同的版本的项目，以便在依赖管理和构建过程正确选择和使用，版本号用标准的数字格式或其他
通过这些属性，Maven坐标提供一种唯一标识的定位项目的方式，在依赖管理中，其他项目可以通过引用该坐标声明对项目的依赖，maven根据这些做标信息，自动下载并引入所需的依赖项

5. maven的生命周期?

阶段 处理 描述
validate 验证项目 验证项目是否正确且所有必须信息时可用的
compile 执行编译 源代码编译在此阶段完成
test 测试 使用适当单元测试框架来测试
package 打包 创建jar/war包，如pom.xml中定义提及的包
verify 检查 检查测试结果，保证质量达标
install 安装 安装打包的项目到本地仓库，以供其他项目使用
deploy 部署 拷贝最终的工程到远程仓库，共享给其他开发人员

6. 熟悉的maven命令？

命令 功能
mvn archetype:generate 创建maven项目
mvn compile 编译源代码
mvn deploy 发布项目
mvn test-compile 编译测试源代码
mvn test 运行应用程序中的单元测试
mvn site 生成项目相关信息的网站
mvn clean 清除项目目录中的生成结果
mvn package 根据项目生成的jar
mvn install 在本地仓库安装jar
mvn eclipse:eclipse 生成eclipse项目文件
mvn jetty:run 启动jetty服务
mvn tomcat:run 启动tomcat服务
mvn clean package -Dmaven.test.skip 清除以前的包重新打包，跳过测试类

7.如何解决依赖传递造成版本冲突？

通过依赖项声明中使用标签排除特定的传递依赖项

8.说说maven的依赖原则？

  1. 最短路径原则（传递依赖的路径越短越好）
  2. pom文件申明顺序优先（路径长度一样，先申明优先）
  3. 覆写原则（当前pom文件申明的直接覆盖父工程传过来的）

9.说说依赖解析机制？

  1. 读取pom文件，包含项目依赖的配置信息
  2. 解析依赖做标，groupId，artifactId，version等，唯一标识了一个依赖项
  3. 检查本地仓库（.m2），是否存在所需依赖项，如果存在且版本匹配，则直接用，无需下载
  4. 下载依赖项，本地仓库没有，或版本不对，尝试从远程仓库下载
  5. 解析依赖关系，检查pom文件，找到他们的传递依赖想，重复以上步骤来解析和下载传递依赖项
  6. 版本冲突解决：有多个依赖项的不同版本时，maven使用冲突解决机制确定最终依赖项，选择路径最短的依赖项或根据pom文件声明顺序优先选
  7. 构建依赖树：根据解析结果构建依赖树，表示项目依赖关系和层次结构，这个依赖树包括直接依赖和传递依赖
  8. 传递依赖管理：将依赖项添加到项目的类路径，以便编译，测试，运行中使用，还会管理传递依赖的范围和冲突解决
